# 精读A07：LLM评估体系

## 0. 一句话定位

**LLM评估是AI产品量产的"质检体系"——没有它，你不知道模型跑到什么水平、能不能上线、有没有跑偏。**

## 1. 这章在讲什么

第14章系统性地介绍了LLM评估的全景图，涵盖三大块：

| 评估维度 | 核心问题 | 对应章节 |
|---------|---------|---------|
| **评估设计** | 测什么、怎么测 | 14.1–14.2 |
| **评估指标** | 如何量化打分 | 14.4–14.6 |
| **评估方法论** | 怎么让打分可靠 | 14.3, 14.7–14.8 |

这章不是在给"哪个模型更好"的答案，而是在教**如何设计一套不会被忽悠的评估体系**。对于PM来说，这是判断AI产品能否上线、迭代方向是否正确的基础能力。

## 2. 评估设计框架

### 2.1 分类体系

作者给出了一个三维分类矩阵：

**维度一：内在评估 vs 外在评估**

- **内在评估（Intrinsic）**：看模型输出本身的质量——比如翻译的BLEU分、代码的Pass@k。便宜、可复现，但不一定反映真实场景表现。
- **外在评估（Extrinsic）**：看模型在实际业务中的效果——比如接入客服LLM后，工单升级率降了多少。贵、慢，但直接回答"这东西有用吗"。

> **三电类比**：内在评估 ≈ 台架测试（测电机功率、扭矩曲线）；外在评估 ≈ 路试（测实际续航、驾驶体验）。台架数据好看不代表路试成绩好。

**维度二：自动评估 vs 人工评估**

| 类型 | 成本 | 速度 | 可复现性 | 有效性 |
|------|------|------|---------|-------|
| 自动（规则型） | 极低 | 极快 | 完美 | 低–中 |
| 自动（模型型） | 低 | 快 | 高 | 中–高 |
| 众包人工 | 中 | 天级 | 中 | 中 |
| 专家人工 | 高 | 周级 | 低–中 | 高 |
| 外在/A/B测试 | 极高 | 月级 | 低 | 极高 |

**维度三：有参考评估 vs 无参考评估**

- **有参考（Reference-Based）**：拿模型输出和标准答案比——BLEU、ROUGE、BERTScore。
- **无参考（Reference-Free）**：没有标准答案时直接评——Perplexity、LLM-as-Judge、人工偏好。

> 开放式对话场景下，不可能穷举所有"正确答案"，必须依赖无参考评估。

### 2.2 什么时候用什么

作者给出了一个PM可以直接用的决策框架：

| 任务类型 | 推荐评估方法 |
|---------|-----------|
| 数学/代码/事实问答（有唯一正确答案） | 精确匹配、执行验证 |
| 开放问答但有参考标准 | 有参考指标兜底 + LLM-as-Judge补充 |
| 主观评价（有用性、语气、创意） | 人工评估 or 校准好的LLM裁判 |
| 多步智能体行为 | 任务成功率 + 轨迹效率 |

**对于三电PM的启示**：AI产品的评估策略应该是分阶段的——开发阶段用自动指标快速迭代，预发布阶段做人工偏好研究，上线后监控业务指标。

## 3. 数据收集与合成数据

### 3.1 人工标注流程

健壮的标注管线包含5个阶段：
1. **任务定义**：标注什么、用什么尺度、什么标准——模糊的定义 → 噪声数据
2. **指南编写**：详细指南 + 边界案例示例 → 小规模试点迭代
3. **标注员招募与培训**：背景匹配 → 校准会议（大家一起标同一批数据，讨论分歧）
4. **质量控制**：插入已知答案的"黄金问题"、随机抽查
5. **统计一致性度量**：用Krippendorff's Alpha、Cohen's Kappa量化一致性

### 3.2 标注者之间的一致性（Inter-Annotator Agreement）

这是评估数据质量的**核心指标**。常用方法：

**Cohen's Kappa（κ）**——两位标注员之间的机会校正一致性：

```
κ = (P₀ - Pₑ) / (1 - Pₑ)

其中：
P₀ = 观察一致率（两位标注员实际一致的百分比）
Pₑ = 偶然一致率（随机猜对的概率）
```

| κ值 | 一致性水平 |
|-----|-----------|
| < 0 | 比随机还差 |
| 0–0.2 | 极低 |
| 0.21–0.40 | 低 |
| 0.41–0.60 | 中等 |
| 0.61–0.80 | 高 |
| 0.81–1.00 | 几乎完美 |

> **三电类比**：就好比两个路试员分别试驾同一辆车，给出驾驶评分。如果两人的评分高度一致（κ高），说明评分标准清晰、评价可信；如果Kappa值低，说明标准模糊或评估维度定义不清——这时候要重新梳理评分标准。

### 3.3 众包 vs 专家标注

| 维度 | 众包 | 专家 |
|------|-----|------|
| 成本 | 低 | 高 |
| 速度 | 快 | 慢 |
| 质量方差 | 高 | 低 |
| 适用场景 | 初筛、大规模数据 | 高精度、领域专业任务 |
| 最佳实践 | 每个任务3–5人投票 | 2人标注+仲裁 |

### 3.4 合成数据生成——当真实数据不够时

核心思想：**用LLM生成训练/评估数据**。三种主要方法：

**Self-Instruct**：种子Prompt → LLM生成更多指令→人工筛选→迭代。低成本扩展指令数据集。

**Evol-Instruct**：从简单指令出发，LLM自己"进化"出更复杂的变体（加约束、加条件、提难度）。

**LLM-as-Judge生成**：用LLM生成带评分的评估样本（合成偏好数据）。但要注意——**合成数据继承源模型的偏差**。

> **三电类比**：合成数据 ≈ 仿真工况数据。通过仿真生成各种路况、驾驶行为数据来训练模型，成本远低于真实路测收集，但仿真和真实世界之间总有差距。

## 4. 排名指标（ELO等）

### 4.1 核心问题：怎么给模型排名？

当你有多个模型/版本，让它们两两PK（人类投票 or LLM投票），最终要给一个**从输赢中算出来的排行榜**。这不是简单的胜率统计，而是要处理"A打败了B，B打败了C，但C打败了A"这种循环套。

### 4.2 ELO评分系统

ELO最早用于国际象棋排名，核心逻辑：

- 每个选手有一个分数（初始通常1200或1000）
- **预期胜率**（Expected Score）：
  ```
  Eₐ = 1 / (1 + 10^((R_b - R_a) / 400))
  ```
  其中R_a是A的分数，R_b是B的分数。分数差400分，高分者胜率约91%。
- **分数更新**（对战之后）：
  ```
  R'_a = R_a + K × (S_a - Eₐ)
  ```
  其中S_a是实际结果（胜=1, 负=0, 平=0.5），K是更新系数（越大变化越快）。

> **三电类比**：ELO评分 ≈ 赛车手积分系统。车手之间比赛，赢了就从对手那里"拿走"积分，输了就"送出"积分。打败强敌加更多分，输给弱旅扣更多分。最终排名反映的是综合对战实力，而不是"某一圈的最快圈速"。

### 4.3 Bradley-Terry（BT）模型

ELO是**在线更新**（每比一场更新一次分数），而BT模型是**批量处理**（用所有历史对战数据一次性算分）。BT模型计算一对物品中A优于B的概率：

```
P(A > B) = β_a / (β_a + β_b)
```

在log空间下等价于使用sigmoid函数的逻辑回归：
```
P(A > B) = σ(θ_a - θ_b)
```

BT模型比ELO更精确（因为利用所有数据），但需要完整对战历史。

### 4.4 Chatbot Arena方法

Chatbot Arena是目前最大的LLM公开排行榜（2024年已超100万人类偏好投票），其方法论：
1. 用户输入Prompt，收到两个匿名模型的回复
2. 用户投票选更好的（或打平）
3. 用BT模型聚合投票 → 生成排行榜
4. Bootstrap置信区间，区间重叠的模型认为是"统计上无法区分"
5. 分数差异超过置信区间的才能说"这个模型更好"

**核心思想：排行榜不是绝对的，要看置信区间。** 如果两个模型的分数区间有重叠，只能说"还没分出高下"。

> **三电类比**：就像多款车型在同一个赛道、同一套计时系统下跑圈，但每辆车跑多次，最终成绩是一个平均值±波动范围。如果A车7.8s±0.2s，B车8.1s±0.3s，你才能说A更快；如果两个区间重叠，那就是"还需更多测试"。

## 5. 生成任务指标

### 5.1 BLEU——翻译任务的"关键词命中率"

BLEU衡量假设翻译和参考翻译之间的**n-gram精度（n-gram precision）**，即"模型输出中有多少词组和标准答案重合"。公式：

```
BLEU = 简短惩罚(BP) × exp(∑ w_n × log p_n)
```

其中p_n是修正后的n-gram精度（避免"过度命中"），BP惩罚过短的输出。

**局限**：
- 设计时假设有**多个参考翻译**（单参考场景下BLEU分很低）
- 不衡量语义相似度——**同义改写会被扣分**
- 对分词敏感

> **三电类比**：BLEU ≈ "翻译准确率=原文和译文的关键词匹配度"。如果原文说"电机效率提高5%"，AI翻译成"电机的效率提升了5个百分点"，关键词匹配但措辞变了，BLEU会扣分。这在学术翻译中可能还行，但**对识别语义正确性很弱**。

### 5.2 ROUGE——摘要任务的"关键词召回率"

ROUGE是**面向召回**的指标族，主要衡量模型摘要有多少内容来自参考摘要：

- **ROUGE-N**：n-gram召回率（ROUGE-1=单词级，ROUGE-2=双词组级）
- **ROUGE-L**：最长公共子序列——衡量句子级结构相似度

**适用场景**：摘要、文档压缩类任务。

### 5.3 BERTScore——用上下文理解算语义相似度

**核心突破**：不再看字面重合，而是用BERT嵌入计算语义余弦相似度。对每个输出token，找参考中最相似的token，取平均。

BERTScore比BLEU/ROUGE**更接近人类判断**，尤其能处理同义改写。

### 5.4 Perplexity——模型"困惑度"

```
PPL = exp(-(1/T) × ∑ log Pθ(w_t | w_{1:t-1}))
```

**越低越好**——表示模型对文本的预测更自信、更准确。

**用途**：相同tokenizer下对比模型、检测分布漂移。
**局限**：不同tokenizer间不可比，和实际质量相关性不一定高。

> PM注意：PPL是开发阶段的"体检查血常规"类指标——低了说明模型没出大问题，但**低PPL不等于高智能**。

### 5.5 Pass@k——代码任务的"试几次能过"

对每个编程问题，模型生成n个解答（通常n=200），计算至少有一个通过所有测试的概率：

```
pass@k = E[ 1 - C(n-c, k) / C(n, k) ]
```

其中c是能通过的解答数，C是组合数。

| k值 | 含义 | 示例(200样本中15个通过) |
|-----|------|----------------------|
| Pass@1 | 一次就对的概率 | 7.5% |
| Pass@10 | 试10次至少一次对的概率 | 53.9% |
| Pass@100 | 试100次至少一次对的概率 | 99.99% |

**启示**：对代码任务，让模型多试几次（增加k值）可以显著提升成功率。在AI编程产品中，"自动重试+选最优"比"一次生成"有效得多。

## 6. 智能体任务评估

智能体（Agent）和纯文本LLM不同——它在环境中执行操作、需要多步推理。评估维度更复杂：

### 6.1 任务成功率（TSR）

最核心的指标：完成任务的百分比。

```
TSR = (完成任务数) / (总任务数)
```

可扩展到带部分分数的版本：TSR_graded（部分完成给部分分）。

### 6.2 轨迹效率（η）

衡量智能体是否"走了弯路"：

```
η = 最优轨迹长度 / 智能体实际轨迹长度
```

η∈(0,1]，η=1表示最优效率。失败任务η=0。

> **三电类比**：导航仪的路径规划效率——目的地是同一个（任务成功），但有的路径多绕了10公里（效率低）。TSR是"是否到达"，轨迹效率是"走得多好"。

### 6.3 工具使用准确率（TUA）

智能体调用外部工具（API、代码解释器、搜索引擎）时，衡量：
- 是否选对了工具
- 参数是否正确
- 调用时机是否合适

### 6.4 SWE-bench——软件工程能力

**真实场景**：给LLM一个GitHub issue描述和代码仓库，让它生成修复补丁。跑测试套件验证。

| 指标 | 值 |
|------|-----|
| 任务数 | 2,294 |
| 最佳开源模型 | ~43% 解决率 |
| 人类表现 | ~87%（15分钟/任务）|
| 评估成本 | ~$0.25/任务 |

**重要认知**：即使是顶级AI编程工具的"底层模型"，在真实软件工程任务上也只有**不到人类一半**的解决率（截至2024年）。这提醒PM：**不要高估AI的代码能力，更不要低估人工review的价值。**

### 6.5 WebArena——网页导航能力

在沙箱浏览器中模拟真实网页操作。812个任务，覆盖电商、论坛、协作开发、内容管理、地图。当前AI表现约35–45%，人类78%。

## 7. LLM-as-Judge方法论

### 7.1 核心思想：用LLM来评LLM

一种**规模化无参考评估**方法。用另一个能力更强的LLM作为"裁判"，按照给定rubric（评分标准）给模型输出打分。

三种常见格式：

| 格式 | 说明 | 适用场景 |
|------|------|---------|
| 逐点评分 | 给一个回复打1-10分 | 单模型质量监控 |
| 配对比较 | 两个回复二选一 | 模型对比、版本迭代 |
| 参考引导 | 给出标准答案再比较 | 事实性任务 |

### 7.2 固有偏差：LLM裁判也是"人"

**位置偏差（Position Bias）**：LLM偏好出现在"前面"或"后面"的回复，偏差幅度可高达10–15个百分点。

缓解策略：
1. **交换增强（Swap Augmentation）**：A vs B和B vs A各评一次，一致即接受、不一致算平
2. **校准提示**：明确告诉裁判"不要被顺序影响"
3. **多裁判团**：用不同模型做评委，取多数投票
4. **思维链条强制**：要求裁判先给出推理再给结论

**冗长偏差（Verbosity Bias）**：LLM裁判倾向于给**更长的回复**更高分，即使多余内容无意义。

> **三电类比**：LLM-as-Judge ≈ AI版的"试驾评价员"。但问题来了：
> - 这个评价员可能**更喜欢先试的车**（位置偏差）
> - 可能**更偏好话多的销售介绍**（冗长偏差）——讲了一大堆但没有干货，反而得分高
> - 所以要用**多个评价员**（多评审团），取综合意见

### 7.3 多评审团（Multi-Judge Panels）

用多个来自不同模型家族的LLM做评委，投票决定结果：

| 投票情况 | 置信度 | 处理方式 |
|----------|--------|---------|
| 3:0（一致） | 高置信度 | 接受 |
| 2:1 | 中置信度 | 接受但标记 |
| 1:1:1（三向平局） | 低置信度 | 需要人工复核 |

评审团一致率 = 评委间一致判定的比例。

### 7.4 G-Eval框架——用概率加权替代硬判决

标准做法是让LLM输出"4分"，**丢弃了模型的不确定性**。G-Eval的改进：
1. 让LLM先生成评估步骤（链式推理）
2. 读取每个分数值（1-5）的log-probability
3. 用概率加权平均算最终得分的**期望值**

```
G-Eval Score = ∑ s × softmax(log P(s))
```

这就像**用后验分布的均值而不是众数**——更稳健、与人类判断的相关性更高。

## 8. 评估陷阱

### 8.1 基准泄露（Benchmark Contamination）

训练数据中**混入了评估集的样本**。导致：模型分数虚高，但实际能力没有提升。

检测方法：
- **n-gram重叠分析**：评估样本和训练语料ROUGE-L>0.8的占比
- **金丝雀字符串**：在评估集中嵌入随机字符串，看模型能否"接上"
- **时间隔离**：使用训练截止日期之后才创建的数据做评估

> **三电类比**：基准泄露 ≈ 把考试题提前给学生做了（分数高不代表真掌握）。模型不是变聪明了，而是**背过标准答案了**。

### 8.2 过拟合到基准

即使没有直接泄露，来回跑同一套基准、反复调参也会导致**隐性过拟合**——模型"学会"了这套测试题的套路。

> **三电类比**：每次测试都用"NEDC工况"跑续航，车企自然会针对NEDC工况做优化——结果NEDC标称续航很好看，实际路跑差一大截。这就是"测试反作用于优化目标"。

### 8.3 Goodhart's Law

> **"当一个指标变成目标，它就不再是好指标了。"**

LLM评估中的三种典型表现：

| 表现形式 | 说明 | 示例 |
|---------|------|------|
| **Reward Hacking** | 模型学会讨好奖励模型而非真正优化 | 变得冗长、自信但事实上错误 |
| **Metric Gaming** | 模型针对BLEU/ROUGE调输出 | 生成更像标准答案但不太有用的内容 |
| **Judge Gaming** | 模型学会LLM裁判的偏好 | 变得话多（利用冗长偏差） |

**防御策略**：
1. 多样化指标——一个指标能被"玩"，同时玩多个指标难得多
2. 保留不用作训练的"隐藏评估"
3. 定期人工抽查
4. 对抗性评估——主动寻找模型弱点
5. 外在验证——定期把内在指标和业务结果做校准

> **三电类比（推荐）**：Goodhart's Law就像"百公里加速"一旦变成唯一考核指标，车企就会把车调成起步窜、但日常驾驶体验很差。指标本身没错，但**只追求这一个指标就会扭曲行为**。关键是多维度评估：加速要测，制动、能耗、操控、舒适性也要测。

### 8.4 其他常见陷阱

| 陷阱 | 说明 |
|------|------|
| **Prompt敏感性** | 评估Prompt的微小变化会导致分数大幅波动 |
| **聚合伪影** | 简单平均不同难度任务的分，掩盖差异 |
| **选择偏差** | 众包标注员不是终端用户的随机样本 |
| **评估-部署错配** | 评估Prompt比真实用户输入"干净"得多 |

**评估设计的终极六问**（每上线一个AI产品前问一遍）：
1. 评估指标和业务目标对齐了吗？
2. 评估数据代表目标分布吗？
3. 评估了泄露和过度拟合风险吗？
4. 所有指标都报置信区间了吗？
5. 评估可复现吗（固定种子、版本化Prompt、公开测试集）？
6. 做过对人工评价或业务结果的外部验证了吗？

## 9. 关键概念速查表

| 概念 | 一句话定义 | 三电类比 |
|------|-----------|---------|
| **ELO** | 基于两两PK输赢的在线排名系统 | 赛车手积分排名 |
| **Bradley-Terry** | 利用全部对战数据批量估算相对实力的模型 | 全年赛事数据统一建模 |
| **Chatbot Arena** | 匿名盲评+BT模型的LLM众包排名系统 | 赛道盲测+专业分 |
| **BLEU** | 翻译任务中n-gram精度（词汇重叠度） | 翻译关键词匹配分 |
| **ROUGE** | 摘要任务中n-gram召回率（内容覆盖度） | 摘要要点覆盖分 |
| **BERTScore** | 用BERT嵌入算语义相似度（不依赖字面重叠） | 理解性判分 |
| **Pass@k** | 代码任务：试k次至少一次通过的概率 | 多路并行试错 |
| **TSR** | 智能体完成任务的比率 | 自动驾驶完成率 |
| **LLM-as-Judge** | 用LLM作为裁判评估LLM输出 | AI试驾评价员 |
| **Swap Augmentation** | AB和BA各评一次来解决位置偏差 | 正反各听一次 |
| **G-Eval** | 读取所有分数的概率分布求期望 | 先列评价维度，再用概率加权 |
| **基准泄露** | 训练数据混入评估集 | 考试题提前给学生做 |
| **Goodhart's Law** | 指标变目标→指标失效 | 只看百公里加速→车变"窜"了 |

## 10. PM视角：怎么评估AI产品

### 10.1 评估体系搭建的四个层次

| 层次 | 是什么 | 成本 | PM做什么 |
|------|-------|------|---------|
| L1 自动指标 | BLEU/ROUGE/Perplexity | 极低 | 了解指标含义和局限，不盲目相信 |
| L2 模型评测 | LLM-as-Judge, Pass@k | 低–中 | 选择评测集、定义评分标准 |
| L3 人工评估 | 专家打分、A/B测试 | 中–高 | 设计标注指南、管理标注流程、监控一致性 |
| L4 业务验证 | 指标落地后的真实效果 | 高 | 把AI指标翻译成业务KPI |

### 10.2 给三电PM的三个实操建议

1. **永远不要只有一个评估指标**
   - 至少3个维度：正确性、效率（耗时/成本）、用户满意度
   - 每个维度至少1个自动指标 + 1个人工/模型指标

2. **建立"评估基准"管理意识**
   - 评估集需要版本管理、定期刷新
   - 每次模型迭代后都跑全量评估
   - 记录每次评估的准确Prompt和配置（否则不可复现）

3. **对"高分"保持怀疑**
   - 分数大涨时，先检查是不是泄露/过拟合
   - 用交换测试消除位置偏差
   - 定期拿新生成的样本做抽查验证

### 10.3 快速检核清单（回答以下问题再上线）

- [ ] 我们在用的评估指标和业务目标之间，建立了映射关系吗？
- [ ] 评估集是否包含训练截止日期之后的新数据？
- [ ] 人工标注员的Kappa值达标了吗（≥0.6）？
- [ ] LLM-as-Judge做了Swap Augmentation去偏差吗？
- [ ] 我们有至少一个"不对训练过程可见"的隐藏评估集吗？
- [ ] 对每个指标都报了置信区间吗？
- [ ] 定期做了外在验证（内在指标 vs 业务结果）吗？

## 11. 记忆钩子

> 一句话记住整章：**ELO排座次，BLEU测翻译，Pass@k验代码，LLM当裁判要看位置偏差，Goodhart说"指标变目标就失效"，PM要永远保持怀疑。**

**三电PM快速记忆法**：
- **评估设计** = 台架测试 vs 路试，各有各的用，别用台架代替路试
- **ELO/BT** = 赛车手积分
- **BLEU/ROUGE** = 你画我猜——猜对几个词
- **BERTScore** = 有脑子的人来打分
- **LLM-as-Judge** = AI试驾员（可能有偏见，多用几个）
- **基准泄露** = 学生偷了标准答案
- **Goodhart's Law** = 考核A，大家就只做A，B必然崩
- **最终原则**：**多样、分层、置信区间、定期验证——评估不是一次性工程，是持续的质量管理。**

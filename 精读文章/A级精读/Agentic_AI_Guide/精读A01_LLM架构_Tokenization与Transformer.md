## 2. LLM 流水线：AI 版的"动力总成"

### 2.1 四阶段总览

原书用一个非常清晰的流水线描述 LLM 怎么处理文本：

```
原始文本 → 分词器 → Token IDs → 嵌入层 → Transformer层(×L) → 词汇概率 → 解码 → 输出文本
                                                          ↕
                                             自回归循环（把输出追加到输入）
```

用三电语言翻译：

| LLM 阶段 | 三电类比 | 说明 |
|----------|---------|------|
| **Tokenization（分词）** | 传感器采样 + ADC转换 | 把连续的模拟信号（自然语言文本）离散化为数字信号（Token ID） |
| **Embedding（嵌入）** | 查特性曲线表 | 每个 Token ID 去查一个映射表，得到对应的"特性向量" |
| **Transformer 层** | 域控制器处理 | 所有 Token 的向量并行经过多层处理，每层都让它们互相"交换信息" |
| **预测解码** | D/A输出 | 把处理结果转回人能读的文本 |

## 3. Tokenization：AI 的"传感器采样"

### 3.1 为什么需要分词

原始文本是人类读的，计算机不认识。必须把文本变成一串数字 ID。

**三种粒度对比**（原书 Table 1.1）：

| 粒度 | 词汇表大小 | 序列长度 | 问题 | 三电类比 |
|------|-----------|---------|------|---------|
| **字符级** | ~256种 | 非常长 | 注意力成本O(n²)；难以学习长距离语义 | 像用1ms采样率测电池电压——数据量巨大但有效信息少 |
| **词级** | ~50万+ | 较短 | 无法处理新词/罕见词；嵌入表巨大 | 像只记录"满电/亏电"两种状态——太粗了，中间状态丢了 |
| **子词级** | 3.2万-12.8万 | 适中 | 最佳权衡 | 像用合理的采样率+量化精度——信息够用，数据量可控 |

**举个栗子**🌰：
- "cryptocurrency" 这个词可能没见过
- 子词分词把它拆成 ["crypt", "ocur", "rency"]，每个都是模型学过的片段
- 就像电池管理系统见到"磷酸铁锂"——虽然没见过这个词，但"磷酸"+"铁"+"锂"都是已知的

### 3.2 BPE（Byte-Pair Encoding）：主流分词算法

原书介绍 BPE 是 GPT、Llama、Mistral 等多数现代 LLM 使用的算法。

**算法逻辑**（超简化版）：
```
1. 从所有单个字符开始（"a", "b", "c", ...）
2. 统计文本中哪两个字符总是挨在一起
3. 把最常挨在一起的合并成一个新符号
4. 重复第2-3步，直到词汇表达到目标大小
```

**三电类比**：
就像你在分析车辆路试数据——一开始记录每个采样点的电压/电流。然后发现"加速+爬坡"这个组合频繁出现，就把它定义为一个新标签"加速爬坡工况"。不断合并频繁出现的组合，最终形成一套高效的工况库。

### 3.3 其他分词方法比较

| 方法 | 代表模型 | 核心思路 | 通俗理解 |
|------|---------|---------|---------|
| **BPE** | GPT-4, Llama-3 | 自底向上，频繁对合并 | 从最小零件开始，把经常一起出现的组装成一个模块 |
| **WordPiece** | BERT | 类似BPE但看信息增益 | 不光看频率，还看"合并后能不能带来新信息" |
| **Unigram LM** | T5, XLNet | 自顶向下，从大词汇表开始剪枝 | 一开始有全部零件，然后去掉最不重要的 |
| **Byte-level BPE** | GPT-2+ | 在原始字节上做BPE | 不依赖任何语言字符集——任何语言、任何符号都能处理 |

### 3.4 关键概念：Fertility（产出率）

**定义**：每个英文词平均对应多少个 token。

- 英语通常 1 个词 ≈ 1.3 个 token
- 一些语言（如日语）可能 1 个词 ≈ 3-4 个 token
- **为什么 PM 需要知道**：产出率直接影响推理成本！比如同样一段话，日语比英语多花 2-3 倍的钱，因为 token 数是按量计费的。

### 3.5 特殊 Token

模型词汇表里有一些不表示语言内容的"控制符号"：

| Token | 作用 | 三电类比 |
|-------|------|---------|
| `<bos>` 开始标记 | 标记序列开始 | 像 CAN 报文的帧头 |
| `<eos>` 结束标记 | 标记序列结束，停止生成 | 像 CAN 报文的帧尾 |
| `<|user|>` 用户标记 | 标识用户发言的开始 | 像区分"请求"和"响应"的 ID |
| `<|assistant|>` 助手标记 | 标识 AI 回复的开始 | 同上 |
| `<pad>` 填充标记 | 把不同长度的句子填充到统一长度 | 像做测试时给不同长度的数据补零对齐 |

## 4. Transformer 架构：AI 的"发动机本体"

### 4.1 从 Encoder-Decoder 到 Decoder-Only

原书详细介绍了 Transformer 的进化：

**原始 Transformer（2017年）**：
```
编码器（Encoder） ←→ 解码器（Decoder）
  双向看输入           逐步生成输出
  (翻译: 读完整句)      (翻译: 逐词生成)
```

就像早期的混合动力系统——电机和发动机两套独立系统配合工作。

**现代 Decoder-Only（GPT风格）**：
```
只有一个解码器
同时负责"理解"和"生成"
```

就像纯电驱动——一套系统同时搞定动力输出和能量回收，更简单、更长续航。

### 4.2 Embedding：查表映射

**核心概念**：把 Token ID（一个整数）映射成一个稠密向量（如 4096 维）。

**通俗理解**：
- 每个词都有一个"特征向量"——一个 4096 个数字组成的列表
- 类似的概念在向量空间中距离近（"king" 和 "queen" 靠近）
- 不相关的概念距离远（"king" 和 "螺丝刀" 很远）

**三电类比**：
就像电池管理系统中的"特性曲线查表"——每个 SOC（荷电状态）对应一组 OCV（开路电压）参数。SOC=50% 时查到的 OCV 值和 SOC=80% 不同。同样，"king" 查到的向量和 "queen" 查到的向量在数值上接近。

**参数规模**：嵌入表 = 词汇表大小 × 维度。
- Llama-3 8B: 128,256 × 4,096 ≈ 5.25 亿参数（占整个模型的 6.5%）

### 4.3 Self-Attention：核心创新

**原书的核心洞察**：Self-Attention 让每个位置都能从所有其他位置"读取"信息。

**通俗理解**：
> Self-Attention 就是让句子里的每个词都能同时"看到"所有其他词，并决定从谁那里吸收多少信息。

**类比——团队会议**：
想象一个产品评审会：
- **Query（查询）**：你当前正在听的内容——"这个功能的 ROI 是多少？"
- **Key（键）**：每个发言人的主题标签——"成本分析"、"用户调研"、"技术方案"
- **Value（值）**：每个发言人的具体内容

你在会议上会做两件事：
1. 你的"Query"去匹配所有人的"Key"——谁说的跟我现在关心的问题相关？
2. 把最相关的人的"Value"加权吸收进来

**Self-Attention 和搜索的关联**（原书精彩比喻）：
> 每个位置都在做一次小型的"搜索"——用自己的 Query 去匹配所有 Key，把最相关 Value 的加权回来。

**这就像产品经理说的"对齐"**——团队里每个人都要和所有人对齐信息，但注意力资源有限，必须决定谁的信息更重要、加权更多。

### 4.4 多头注意力

**概念**：不是只有一个注意力机制，而是并行运行多个（如 32 个），每个"头"关注不同的关系。

**通俗理解**：
- 一个头关注"语法关系"（哪个词修饰哪个词）
- 一个头关注"语义关系"（这个词指代什么）
- 一个头关注"位置关系"（距离多远）
- ...

**三电类比**：
就像 BMS 同时监控多个维度：
- 一个模块监控电压
- 一个模块监控温度
- 一个模块监控电流
- 最后综合所有信息判断电池状态

### 4.5 位置编码

Transformer 没有"顺序"概念——它一次看到所有词，不知道哪个在前哪个在后。位置编码就是给每个词加上"位置标签"。

**通俗理解**：就像给团队成员编号"1号位、2号位……"——不然并行处理时不知道"先点火"和"后点火"的区别。

**RoPE（旋转位置编码）**：现代最主流的方法。用一个"旋转"操作来编码位置——位置越远的词，向量旋转角度越大。就像每个词都有一个"位置仪表盘"，指针角度表示了它在句子中的位置。

### 4.6 Feed-Forward Network（FFN / MLP）

Self-Attention 之后，每个位置还要经过一个"前馈网络"——一个简单的两三层神经网络。

**通俗理解**：Self-Attention 负责"串行交换信息"，FFN 负责"独立深度思考"。

- Self-Attention：团队讨论，信息共享
- FFN：各回各工位，独立消化、加工

**三电类比**：
- Self-Attention = CAN 总线上的数据交换
- FFN = 每个 ECU 独立处理自己收到的数据

### 4.7 层归一化

**概念**：每个子层（Attention 或 FFN）的输出做一次"标准化"——把数据拉回到一个稳定的范围内。

**通俗理解**：就像整车标定时做的"归一化处理"——不管输入信号多大范围，都归一化到 0-1 之间，保证后续处理稳定。

**Pre-Norm vs Post-Norm**（原书重点）：
- **原始 Transformer**：Post-Norm（做完计算再归一化）
- **现代 LLM（GPT-3, Llama, Mistral）**：Pre-Norm（先归一化再做计算）
- Pre-Norm 训练更稳定，不需要特殊的学习率预热策略

## 5. 重要概念速查表（PM 版）

| 概念 | 一句话理解 | 为什么 PM 需要关心 |
|------|-----------|------------------|
| **Tokenization** | 把文字切成小块 | 直接影响推理成本——中文比英文"贵" |
| **BPE** | 把常见组合打包成新词 | 决定模型能多好地处理专业术语 |
| **Vocabulary Size** | 词汇表有多大 | 大词汇表效果好但更贵 |
| **Embedding** | 给每个词一个"特征向量" | 影响模型理解语义的质量 |
| **Self-Attention** | 所有词互相"看"一遍 | 计算复杂度O(n²)，序列长度翻倍成本翻4倍 |
| **Multi-Head Attention** | 多个"视角"并行看 | 增加模型容量但算力也增加 |
| **Positional Encoding** | 给词标"位置" | 没有这个 Transformer 就不懂语序 |
| **FFN / MLP** | "独立思考"层 | 占总参数的 2/3，是参数大头 |
| **Layer Normalization** | 归一化稳定训练 | 决定训练能不能收敛 |
| **Causal Mask / 因果掩码** | 防止"看到未来" | 保证生成时只依赖已生成的词 |

## 6. 成本直觉

### 6.1 Attention 的 O(n²) 复杂度

这是 PM 最需要记住的数字：
- 序列长度 n → 注意力计算量 ∝ n²
- 若上下文从 4K 扩大到 32K → 计算量增加 64 倍！
- 这就是为什么长上下文模型那么贵

**三电类比**：就像电机设计中的"齿槽转矩分析"——齿数翻倍，计算量翻4倍。不是线性增长，而是指数级增长。

### 6.2 Embedding 表的大小

| 模型 | 词汇表 | 维度 | Embedding参数量 |
|------|--------|------|----------------|
| GPT-2 | 50,257 | 768 | 3860万 |
| Llama-3 8B | 128,256 | 4,096 | 5.25亿 |
| GPT-3 175B | 50,257 | 12,288 | 6.2亿 |

Embedding 表只占总参数的一小部分（Llama-3 中占 6.5%），但调用频率极高——每个 token 都要查一次。

## 8. 产品决策启发

### 8.1 Tokenization 选型影响

- **词汇表大小影响多语言性能**：如果产品面向全球市场，词汇表越大越好（Llama-3 用了 128K）
- **Fertility 影响成本**：日语/中文的每词 Token 数高于英语，同样功能的"每对话成本"不同

### 8.2 Attention 复杂度影响上下文窗口设计

- 要支持长上下文（如阅读 100 页文档），Attention 成本会暴涨
- 这就是为什么需要 Flash Attention（下篇详解）——它把 O(n²)的实际运行时间大幅降低

### 8.3 Decoder-Only 是主流选择

如果需要在产品和商业层面做技术选型：
- **Decoder-Only**：绝大多数场景的首选（ChatGPT、Claude、Llama）
- **Encoder-Decoder**：翻译、摘要等输入输出结构差异大的任务（较少见）
- **Encoder-Only**：只做文本分类/检索（如 BERT）

# 精读A08：RAG（检索增强生成）与智能体导论

## 0. 一句话定位

**从"聊天机器人的单次问答"到"自主智能体的循环决策"，RAG（检索增强生成）是智能体获取外部知识的"感官系统"——让AI学会"查手册"而不是"全靠记忆力"。**

---

## 1. 智能体AI概述

### 1.1 从聊天机器人到自主智能体的光谱

传统聊天机器人的工作方式是**单轮问答**：用户输入问题，LLM输出答案，对话结束。而**Agentic AI（智能体式AI）** 的核心区别在于：**LLM在一个持续的循环中运作**——它接收来自环境的观察（用户消息、工具输出、传感器数据），推理下一步行动，执行行动（调用工具、执行代码、发送API请求），然后迭代直至目标达成或主动请求人工介入。

本书（Chapter 15）将智能体的演进分解为以下几个关键挑战，这些挑战正是后续各章要解决的核心问题：

| 挑战 | 说明 | 对应章节 |
|------|------|----------|
| **持久性（Persistence）** | 智能体必须记住它做过什么、哪些失败了、建立了什么上下文——跨轮次、跨会话甚至跨天 | Ch17 Memory |
| **接地性（Grounding）** | 智能体必须访问最新的、领域特定的、训练数据中不存在的知识 | **Ch16 RAG** |
| **行动（Action）** | 智能体必须通过明确定义的接口与外部系统（数据库、API、文件系统、浏览器）交互 | Ch21 MCP |
| **协调（Coordination）** | 复杂任务超过单个智能体的能力，需要多智能体协作、委派和协商 | Ch24 Multi-Agent |
| **安全性（Safety）** | 自主行动需要护栏、人类监督和优雅降级 | Ch18 Harness |

### 1.2 智能体的分层架构

生产级智能体系统被构建为**分层架构**，从底层到顶层依次为：

## 2. RAG的动机与场景选择

### 2.1 参数化知识 vs 非参数化知识

| 概念 | 三电类比 | 说明 |
|------|----------|------|
| **参数化知识** | 工程师头脑里记住的知识 | LLM在训练时学习的权重参数中存储的知识。优点是推理速度快，缺点是：
  - **静态**：训练完成后知识无法更新
  - **不完整**：训练数据中不包含专有/最新信息
  - **易幻觉**：对不确定的知识会"编造" |
| **非参数化知识** | 外部资料库/手册/标准文档 | 存储在外部数据库、文档、API中的知识。RAG通过在推理时检索这些外部知识来补足LLM的不足。 |

### 2.2 RAG vs 微调 vs 长上下文的选择

这是RAG系统设计中最关键的决策之一。本书给出了以下对比：

| 维度 | RAG | 微调（Fine-Tuning） | 长上下文（Long Context） |
|------|-----|---------------------|-------------------------|
| **知识更新** | 即插即用，更新语料库即可 | 需要重新训练 | 不需要（但受窗口限制） |
| **幻觉控制** | 好——生成基于检索到的内容 | 差——依赖模型记忆 | 中等——依赖Prompt工程 |
| **领域适配** | 中等——取决于检索质量 | 好——模型学会领域风格 | 差——仅有上下文引导 |
| **延迟** | 中等（检索+生成） | 低（纯生成） | 低（纯生成） |
| **成本** | 中等 | 高（训练成本） | 中等（长上下文token成本） |
| **复杂度** | 中等 | 高 | 低 |

**PM决策指南**：
- **需要频繁更新知识**（如政策文档、产品规格变更）→ **RAG**
- **需要模型学会特定写作风格或格式**（如技术报告模板）→ **微调**
- **一次性处理超长文档**（如整个技术手册）→ **长上下文**
- **最优方案**：RAG + 微调结合（详见第16.10节RAFT）

> 三电类比：RAG = 让AI学会"查手册"而不是"全靠记忆力"。微调 = 送工程师去标准培训。长上下文 = 给工程师一本超厚手册现场翻——但翻得慢、翻不全。

### 2.3 RAG + 微调协同（RAFT）

RAFT（Retrieval-Augmented Fine-Tuning）是一个关键创新：**同时使用相关文档和干扰文档训练模型**，教会模型识别和使用正确的上下文。训练时，模型看到的是（问题、正确答案、相关文档、k-1个干扰文档）的混合，通过Chain-of-Thought从正确文档中引用来生成答案。

这与三电场景的联系：当工程师被训练为"在查阅BMS系统手册时，忽略空调系统手册中的无关信息"，这就是RAFT的效果。

## 3. RAG核心架构

### 3.1 标准三阶段流水线

```
                   ┌────────────┐
                   │   索引     │
                   │  (Index)   │
                   └─────┬──────┘
                         │ 文档分块→向量化→存储
                         ▼
                   ┌────────────┐
   查询───────▶  │   检索     │
                   │ (Retrieve) │
                   └─────┬──────┘
                         │ 相似度搜索→召回top-k
                         ▼
                   ┌────────────┐
                   │   生成     │
                   │ (Generate) │
                   └─────┬──────┘
                         │ 上下文注入→LLM生成
                         ▼
                     回答
```

**阶段详解**：

**1. 索引（Index）**
- **分块（Chunking）**：将长文档切分成可检索的段落
- **向量化（Embedding）**：用嵌入模型将每个段落转换为向量
- **存储（Storage）**：存入向量数据库

**2. 检索（Retrieve）**
- 将用户查询也转为向量
- 在向量数据库中做近似最近邻搜索（ANN）
- 召回最相关的top-k个文档段落

**3. 生成（Generate）**
- 将检索到的文档作为上下文注入LLM的Prompt
- LLM基于检索内容+原始查询生成回答
- 要求LLM引用检索来源

### 3.2 关键的设计变量

| 变量 | 选择范围 | 影响 |
|------|----------|------|
| 分块大小 | 128-1024 tokens | 过小丢失上下文，过大引入噪声 |
| Top-k数量 | 3-20 | 越多召回率越高但噪声越大 |
| 检索方法 | 稀疏/稠密/混合 | 影响召回质量和延迟 |
| 嵌入模型 | BGE/OpenAI/Voyage等 | 决定检索质量上限 |

## 4. 检索方法全览

### 4.1 方法对比总表

| 方法 | 类型 | 原理 | 三电类比 | 优势 | 劣势 |
|------|------|------|----------|------|------|
| **BM25** | 稀疏检索 | 基于词频-逆文档频率（TF-IDF的增强版）的关键词匹配 | 用关键词查手册（搜索"电池过热"只会找到含这些字的段落） | 简单、可解释、无需GPU | 词义鸿沟——"电池温度过高"匹配不到"热逃逸" |
| **DPR** | 稠密检索 | 用双编码器分别编码查询和文档，计算语义相似度 | 按"意思"查手册（"电池温度过高"能查到"热管理系统"） | 语义理解强 | 需要训练数据、领域迁移差、冷启动问题 |
| **混合检索** | 混合 | BM25 + Dense，用RRF（倒数排名融合）合并结果 | 既查关键词又查意思，综合排名 | 最佳实践标配 | 需要维护两套索引 |
| **SPLADE** | 学习型稀疏 | 用Transformer学习稀疏词权重，结合稠密语义+稀疏可解释 | 既能理解意思，又告诉你"为什么查到这篇" | 可解释性+语义理解 | 推理成本高于BM25 |
| **ColBERT** | 后期交互 | 分别编码查询和文档的每个token，最后阶段做细粒度匹配 | 查手册时，把问题里的每个词和手册的每个句子逐个比对 | 细粒度匹配、延迟友好 | 索引存储开销大 |

### 4.2 关键检索公式

**BM25得分**：
```
score(q, d) = Σ(t in q) IDF(t) · TF(t, d) · (k₁ + 1) / (TF(t, d) + k₁ · (1 - b + b · |d|/avgdl))
```

**DPR**：使用两个BERT编码器：
```
P(d|q) = exp(E_Q(q) · E_D(d)ᵀ) / Σexp(E_Q(q) · E_D(d')ᵀ)
```

### 4.3 混合检索（Hybrid Search）

当代RAG系统的默认选择。步骤：
1. 用BM25做关键词检索 → 得到排名列表A
2. 用向量相似度做语义检索 → 得到排名列表B
3. 用**RRF（Reciprocal Rank Fusion）** 融合：
   ```
   RRF_score(d) = 1/(k + rank_A(d)) + 1/(k + rank_B(d))
   ```
4. 按融合得分重新排序

> 三电类比：混合检索 = 查手册时既用目录索引（BM25），又用"意思联想"（DPR），综合两个结果找到最相关的页面。

## 5. Chunking策略

### 5.1 为什么分块重要？

分块是RAG系统中最容易被忽视但影响最大的环节。**不良的分块会直接限制检索质量的边界**——再好的检索器也无法找回一个被切碎/切乱的段落。

### 5.2 主流策略对比

| 策略 | 原理 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|----------|
| **固定大小分块** | 按固定token数（如256/512）切割，带重叠 | 简单、可预测 | 可能切断语义完整的段落（如把"电池温度超过60°C"和"需立即降功率"分到两段） | 快速原型、内容结构不明确的文档 |
| **语义分块** | 检测语义边界（主题切换点）切分 | 保留语义完整性 | 计算开销大 | 内容主题明确、章节清晰的文档 |
| **文档结构感知** | 按文档层级（章节/段落/句子）切分 | 保留逻辑结构 | 依赖文档格式质量 | 结构良好的PDF/HTML文档 |
| **父子分块** | 大块（父）包含上下文，小块（子）检索，检索到子块后返回父块 | 兼顾精确检索和丰富上下文 | 存储开销大 | 需要同时支持精确定位和全局理解的场景 |

### 5.3 最佳实践

- **重叠（Overlap）**：10-15%的字符/Token重叠，防止信息在切分边界丢失
- **元数据标注**：每个chunk附带来源、日期、章节信息，支持后续预过滤（pre-filtering）
- **动态分块**：根据文档类型自适应选择分块策略

> 三电类比：Chunking策略 = 怎么把厚厚的技术手册切成合适的段落。固定大小 = 每10页一刀切，可能把"电池安全规范"的第1页和第2页分到不同文件夹；语义分块 = 按"热管理""BMS""充电策略"这样的大主题切；父子分块 = 先切成大节，每节再细分成可检索的小段，找到小段后返回整节的内容以便理解上下文。

## 6. 高级RAG模式

### 6.1 模式对比总览

| 模式 | 核心思想 | 何时使用 | 复杂度 |
|------|----------|----------|--------|
| **查询变换** | 改写/分解用户查询以适应检索系统 | 查询措辞模糊或过于复杂 | 低 |
| **重排序** | 用交叉编码器重新精排初检结果 | 需要提高结果精度 | 中 |
| **上下文压缩** | 对检索结果进行过滤/压缩 | 检索结果包含大量噪声 | 中 |
| **Self-RAG** | 在生成过程中自我判断是否需要检索、检索结果是否相关 | 需要选择性检索 | 高 |
| **CRAG** | 对检索结果做纠错性验证和修正 | 检索源不可靠 | 高 |
| **Adaptive RAG** | 根据查询复杂度动态选择检索策略 | 查询多样性大 | 中高 |
| **Graph RAG** | 构建知识图谱，支持全局汇总查询 | 需要跨文档全局分析 | 很高 |
| **RAG-Fusion** | 生成多个角度的查询，综合多源结果 | 查询需要多维度信息 | 中 |

### 6.2 各模式详解

#### 查询变换（Query Transformation）
- **Query Rewriting**：将模糊查询改写得更适合检索（如"那个电池问题"→ "2025年Q3电池安全事故复盘"）
- **Query Decomposition**：将复杂多跳查询分解为子问题（如"电池成本变化趋势及其对整车售价的影响"→ "近3年电池成本变化"+"电池成本变化如何传递到售价"）
- **HyDE（Hypothetical Document Embeddings）**：先生成一个假设性的理想文档，用它的向量去检索

#### 重排序（Re-Ranking）
- 第一阶段：用轻量级检索器（BM25/DPR）快速召回Top-50～Top-100
- 第二阶段：用**交叉编码器（Cross-Encoder）** 逐对计算查询-文档相关性分数，重新排序
- 交叉编码器比双编码器更精确，但计算量更大——所以只用于第二阶段

> 三电类比：重排序 = 先快速从手册堆里抽50页相关页（用关键词），再仔细逐页判断哪些最相关（读一遍内容），最后只留Top-5给工程师看。

#### Self-RAG
Self-RAG让模型在生成过程中通过**反射标记（reflection tokens）** 自主决定：
1. **是否需要检索？**（IsRetrieve）
2. **检索结果是否相关？**（IsRelevant）
3. **生成的内容是否被检索结果支持？**（IsSupport）

这实现了"选择性检索"——只有在模型不确定时才去查手册，而不是每次都要查。

#### CRAG（Corrective RAG）
CRAG引入了一个**纠错机制**：当检索结果不可靠时，不是简单放弃，而是：
- **确认正确**：正常生成
- **部分相关**：保留相关部分，放弃不相关部分
- **完全不相关**：回退到Web搜索等替代方案

#### Graph RAG
Graph RAG在文档基础上构建知识图谱：
1. 从文档中提取实体和关系
2. 构建基于社区的摘要（Community Summaries）
3. 对全局性问题（如"电池技术的整体发展趋势是什么"）进行摘要级检索而非片段级检索

#### RAG-Fusion
RAG-Fusion的核心创新是**多查询生成**：
1. 用户输入一个查询
2. 用LLM生成多个角度不同的子查询
3. 对每个子查询分别检索
4. 用RRF融合所有结果

> 三电类比：RAG-Fusion = 查多本手册，综合多源信息回答。不只是查"电池手册"，还查"整车手册""安全标准""供应链报告"，综合所有来源做出回答，避免单一来源的片面性。

## 7. Agentic RAG

### 7.1 超越静态检索

传统RAG是**静态流水线**：查询 → 检索 → 生成，一步到位。Agentic RAG的核心区别是：**将检索解构成一系列智能体决策**——智能体可以决定查什么、去哪里查、要不要再查。

```
      ┌─────────┐
      │  计划    │ ← 分解查询为子问题
      └────┬────┘
           ▼
      ┌─────────┐
      │  路由    │ ← 决定去哪个数据源查
      └────┬────┘
           ▼
      ┌─────────┐
      │  检索    │ ← 执行检索
      └────┬────┘
           ▼
      ┌─────────┐     否
      │  评估    │ ──────→ 返回检索（改进查询）
      └────┬────┘
           │ 是
           ▼
      ┌─────────┐
      │  生成    │
      └─────────┘
```

### 7.2 多源路由（Multi-Source Routing）

Agentic RAG可以根据查询的语义决定去哪个数据源检索：

```python
def route_query(query: str) -> str:
    """Route query to appropriate data source."""
    # 分类查询到不同源
    if "specification" in query.lower():
        return "product_docs"      # 产品规格文档
    elif "2025" in query or "2026" in query:
        return "quarterly_reports" # 季度报告
    elif any(code in query for code in ["ISO", "GB/T", "SAE"]):
        return "standards_library" # 标准文档库
    else:
        return "general_kb"        # 通用知识库
```

### 7.3 工具增强型RAG

Agentic RAG可以组合多种工具来解决单一检索无法处理的问题：

| 工具 | 功能 | 三电场景举例 |
|------|------|-------------|
| 文档检索 | 检索内部知识库 | 查电池规格手册 |
| SQL查询 | 查结构化数据库 | 查近期电池不良率数据 |
| Web搜索 | 查互联网信息 | 查竞争对手最新电池技术 |
| Python执行 | 做计算 | 计算电池包能量密度 |

### 7.4 Search-R1：RL训练的Agentic RAG

这是本书介绍的前沿方法（第16.7.6节）。关键创新：

**传统RAG问**："给定这个查询，我应该检索什么？"（管道决策）
**Search-R1问**："基于我目前推理到的内容，我还需要更多信息吗？如果需要，具体什么信息？"（生成过程中的学习决策）

Search-R1在推理过程中可以发出特殊token `<search>query</search>` 触发实时检索，检索结果注入到推理上下文后继续生成。整个轨迹（推理+搜索+最终答案）通过强化学习（GRPO）优化。

**结果**：一个7B模型的Search-R1在开放域QA上：
- 优于标准RAG 15-20%
- 优于Prompted Agentic RAG 8-12%
- 接近70B模型+标准RAG的性能

> 三电类比：Search-R1 = 一个工程师写分析报告时，不先查完所有资料再写，而是边写边想"这块信息不够确定"，即时去查具体问题，查完继续写。**不是考前突击查全书，而是边做题边翻特定知识点。**

## 8. RAG评估

### 8.1 三层评估体系

RAG的评估比单独评估检索或生成更难，因为错误可能发生在任何阶段并相互叠加：

```
第一层：检索质量
  ├── 召回率（Recall@K）：相关文档被召回了多少
  ├── 精确率（Precision@K）：召回的结果中有多少是相关的
  ├── MRR（Mean Reciprocal Rank）：第一个相关结果排第几
  └── NDCG（Normalized Discounted CG）：考虑排序位置的加权指标

第二层：生成质量
  ├── 忠实度（Faithfulness）：回答中的每个声明是否都能追溯到检索文档
  ├── 答案相关性（Answer Relevance）：回答是否回答了问题
  ├── 上下文精确率（Context Precision）：检索上下文有多精确
  └── 上下文召回率（Context Recall）：真实答案所需信息是否都在检索上下文中

第三层：端到端质量
  ├── 用户偏好
  ├── 任务成功率
  └── 延迟调整效用
```

### 8.2 关键指标公式

**召回率@K**：
```
Recall@K = |R_K ∩ R| / |R|
```
其中R_K是top-K检索结果集合，R是全部相关文档集合。

**MRR**：
```
MRR = (1/|Q|) · Σ(1/rank_i)
```
rank_i是第i个查询第一个相关结果的排名。

**忠实度**：
```
Faithfulness = (# 被检索文档支持的声明) / (# 回答中的总声明数)
```

### 8.3 RAGAs框架

RAGAs（Retrieval Augmented Generation Assessment）提供了一个**无需人工标注**的评估框架：

```python
from ragas import evaluate
from ragas.metrics import (
    faithfulness, answer_relevancy,
    context_precision, context_recall,
    answer_correctness,
)

results = evaluate(
    dataset=eval_dataset,
    metrics=[
        faithfulness, answer_relevancy,
        context_precision, context_recall,
        answer_correctness,
    ],
)
```

### 8.4 常见失败模式

| 失败模式 | 现象 | 原因 | 排查方向 |
|----------|------|------|----------|
| **检索缺失** | 相关文档存在但未召回 | 分块不当、嵌入模型不匹配、查询-文档词汇鸿沟 | 检查分块策略、嵌入模型选择 |
| **上下文污染** | 检索到的文档包含误导信息 | 知识库质量差、检索范围过大 | 加质量过滤、缩小检索范围 |
| **中间丢失** | LLM忽略长上下文中间的重要信息 | LLM对长上下文的首尾注意力更强 | 重排检索结果（最相关放在首尾） |
| **过度检索** | 太多chunk稀释了有效信号 | Top-k设置过大 | 减少Top-k、加后过滤 |
| **检索条件下幻觉** | 模型忽略检索内容，凭记忆生成 | 预训练数据与检索内容矛盾 | 增强Prompt约束、加Fine-tuning |
| **引用伪造** | 模型声称某个文档支持了某个声明 | LLM合成了不存在的引用 | 强制引用验证、加Citation检查 |

## 9. 关键概念速查表

| 概念 | 一句话定义 | 三电类比 |
|------|-----------|----------|
| **RAG** | 检索增强生成——在推理时检索外部知识辅助LLM生成 | 让AI学会"查手册"而不是"全靠记忆力" |
| **参数化知识** | LLM权重中存储的知识（训练学到） | 工程师头脑里记住的知识 |
| **非参数化知识** | 外部数据库中的知识（检索获得） | 外部资料库/手册/标准文档 |
| **BM25** | 基于关键词匹配的稀疏检索 | 用关键词查手册的目录索引 |
| **DPR** | 基于语义相似度的稠密检索 | 按"意思"查手册 |
| **混合检索** | BM25 + Dense融合 | 既查关键词又查意思 |
| **RRF** | 倒数排名融合——组合多个排序结果 | 综合多本手册的目录排名 |
| **ColBERT** | 后期交互——查询和文档的token级细粒度匹配 | 逐词比对问题与手册内容 |
| **Cross-Encoder** | 查询-文档对做精细相关度评分 | 仔细阅读判断两段文字是否相关 |
| **HyDE** | 生成假设文档并用其向量检索 | 先想好"理想的答案长什么样"，再按样子找 |
| **Chunking** | 将长文档切成可检索的段落 | 把厚技术手册切分成合适的段落 |
| **父子分块** | 大块存上下文、小块做检索 | 手册目录页定位到小节，取出整章 |
| **Self-RAG** | 模型自主判断是否需要检索 | 工程师不确定时才去翻手册 |
| **CRAG** | 对检索结果纠错验证 | 翻到的手册页有不一致的地方，多方验证 |
| **Agentic RAG** | 智能体根据场景自主决定检索策略 | 一本手册查不到，主动去查另一本 |
| **RAG-Fusion** | 多查询生成 + 多源融合 | 查多本手册，综合多源信息回答 |
| **Graph RAG** | 构建知识图谱支持全局分析 | 不只是查手册，还整理出手册中的知识图谱 |
| **Search-R1** | 用RL训练模型学会何时搜索、搜索什么 | 边推理边反思"这里不确定，需要查一下" |
| **RAFT** | 用干扰文档训练模型使用正确上下文 | 训练工程师在众多手册中识别正确的那个 |
| **Faithfulness** | 回答是否忠实于检索到的上下文 | 回答的内容是否真的在"查到的资料"里 |
| **RAGAs** | 无参考答案的RAG评估框架 | 自动检测回答质量 |
| **MCP** | 模型上下文协议——工具集成标准 | 智能体的"USB接口" |
| **ReAct** | 推理+行动交错——Agent的核心设计模式 | 边想边做 |
| **A2A** | 智能体间通信协议 | 多个"工程师角色"之间的协作规范 |

## 10. PM视角：什么时候用RAG

### 10.1 决策树

```
你的场景需要AI回答/处理什么？
│
├── 静态知识、高频不变的内容
│   └── 用 Fine-tuning（如：标准报告格式、专业术语）
│
├── 一次性处理整份大文档
│   └── 用 Long Context（如：单个技术方案评审）
│
├── 需要实时/周期性更新的内容
│   └── ▶ 用 RAG ◀
│       ├── 简单查询 → 基础RAG
│       ├── 复杂多跳查询 → Agentic RAG
│       ├── 分析趋势/全局模式 → Graph RAG
│       └── 跨多手册综合 → RAG-Fusion
│
└── 需要特定格式+知识更新
    └── RAG + Fine-tuning 组合（RAFT）
```

### 10.2 整车三电场景的典型应用

| 场景 | 推荐方案 | 说明 |
|------|----------|------|
| "查询某车型电池包规格" | 基础RAG | 直接在规格文档库中检索 |
| "对比我们和竞争对手的电池能量密度趋势" | RAG-Fusion | 多源查询：内部数据+行业报告+公开信息 |
| "这个项目的技术方案涉及哪些安全标准？" | Agentic RAG | 需要路由到不同的标准文档库（ISO/GB/SAE） |
| "总结近两年电芯技术路线的演变" | Graph RAG | 需要从多份报告、论文、专利中提取实体关系 |
| "分析这个BMS故障报告的可能原因" | Agentic RAG + Tools | 检索相似故障案例+查数据库+做计算验证 |

### 10.3 RAG系统设计的关键问题

1. **查询分布是什么？** 事实型 vs 分析型 vs 多跳型，需要不同的检索策略
2. **语料库有多大和多动态？** 百万级+频繁更新 → 需要Managed向量数据库 + 增量索引
3. **延迟要求多高？** 亚100ms → 不能做Re-ranking和Agent循环；批处理可以
4. **接地性多关键？** 高安全领域（医疗、法律、车辆安全）→ 必须做Faithfulness评估和引用验证
5. **词汇是否专业？** 领域术语 → 可能需要混合检索或领域适配的嵌入模型

### 10.4 RAG最佳实践速查

- **从简单开始**：好的分块 + 基础RAG常胜过复杂的系统 + 差的分块
- **先评估检索**：修复检索瓶颈后再优化生成
- **混合检索是默认选择**：BM25 + Dense + RRF
- **加Re-ranking**：交叉编码器重排Top-20 → 高性价比
- **监控Faithfulness**：用LLM Judge在生产环境中跟踪幻觉率
- **缓存嵌入**：文档嵌入一次；频繁查询嵌入也缓存
- **分块加重叠**：10-15%重叠防止边界信息丢失
- **存丰富元数据**：来源、日期、章节、文档类型 → 支持预过滤
